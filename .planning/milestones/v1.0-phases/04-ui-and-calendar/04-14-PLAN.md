<objective>
Close UAT GAP 1: after signing in, the app redirects back to /login instead of showing the navigation shell (Test 5, major). The AuthGate (router.tsx) redirects to /login whenever `session` is null. The Supabase client (`supabase.ts`) is created with NO explicit auth options, and `useAuth` relies solely on `onAuthStateChange` without seeding the store from an initial `getSession()`. The result is a window where AuthGate reads a null session and bounces the user to /login before (or instead of) the session being restored.

Purpose: A user who has just authenticated cannot reach the app. This blocks every authenticated screen behind the gate (UI-07 navigation shell and all destinations).

Output: The Supabase client explicitly enables `persistSession`, `autoRefreshToken`, and `detectSessionInUrl`. `useAuth` seeds the store from `getSession()` on mount and keeps `isLoading` true until the initial session is resolved, and the `onAuthStateChange` handler never overwrites a real session with a transient null. AuthCallbackScreen's diagnostic console logs are removed.
</objective>

<task type="auto">
  <name>Task 2: Seed the auth store from getSession on mount and guard against null overwrite (UAT GAP 1)</name>
  <files>frontend/src/hooks/useAuth.ts</files>
  <read_first>
    - frontend/src/hooks/useAuth.ts (current state: useEffect subscribes ONLY to supabase.auth.onAuthStateChange and sets the store from each event with isLoading false; there is no initial getSession() call, so the very first authenticated render can occur before the INITIAL_SESSION event arrives — and a null newSession from a transient event would flip the store to signed-out)
    - frontend/src/stores/authStore.ts (setAuth replaces session, user, and isLoading wholesale; isLoading starts true)
    - frontend/src/router.tsx lines 46-69 (AuthGate: while isLoading it shows a spinner; when not loading and session is null it Navigates to /login — so isLoading must stay true until the initial session is resolved, and session must not be transiently nulled)
  </read_first>
  <action>
    Make the auth hook resolve an initial session before AuthGate decides, and prevent a transient null event from signing the user out.

    Concretely, in the useEffect of useAuth:
    - Add an initial session resolution: call supabase.auth.getSession() once on mount; when it resolves, call setAuth with the returned session, its user (session?.user ?? null), and isLoading: false. Guard this against running after unmount using a local `let active = true` flag checked before calling setAuth (set active = false in the cleanup).
    - Keep the existing onAuthStateChange subscription, but change its handler so it does NOT overwrite an existing non-null session with a null one EXCEPT on an explicit SIGNED_OUT event. Concretely: in the callback, when newSession is null and the event is not 'SIGNED_OUT', return without calling setAuth (this ignores transient INITIAL_SESSION-with-null races that would otherwise bounce a just-authenticated user). Otherwise call setAuth with the new session, its user, and isLoading false as before. The event name is the first argument of the onAuthStateChange callback (currently named `_event`; rename to `event` so it can be inspected).
    - Preserve the cleanup that unsubscribes the subscription, and add the active-flag reset there.

    Rationale: The redirect loop happens because AuthGate reads a null session before the session is established, or because a transient null event clobbers a good session. Seeding from getSession() resolves the real session up front; guarding null overwrites prevents the clobber. SIGNED_OUT remains the only path that clears the session (LoginScreen/Settings sign-out already rely on it).

    Do not change the hook's return shape ({ session, user, isLoading }) or its single-call-near-root usage in RootProvider.
  </action>

    - An unmount guard (active flag) prevents setAuth after cleanup
    - The onAuthStateChange handler ignores a null newSession unless the event is SIGNED_OUT
    - SIGNED_OUT still clears the session in the store
    - The hook still returns { session, user, isLoading } and is still called once in RootProvider
    - npm run build succeeds and the vitest suite shows no NEW failures introduced by this change
  </acceptance_criteria>
  <done>On sign-in the store is populated with the real session before AuthGate evaluates, and transient null events no longer bounce the user; the app lands on the navigation shell instead of looping to /login.</done>
