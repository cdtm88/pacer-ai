<objective>
Close UAT GAP 5: sending a chat message returns no usable response. Root cause (confirmed in code): the frontend `ChatScreen.handleSend` builds the SSE URL as `/chat/stream?conversation_id=...&message=<text>`, but `chat_stream` in `api/routes/chat.py` only declares a `conversation_id` query param — it never reads `message`. It loads conversation history (empty for a fresh conversation), seeds the hardcoded `_OPENING_MESSAGE`, and streams a reply to THAT instead of the user's actual message. Additionally, chat turns are never persisted (chat.py docstring explicitly defers this), so there is no continuity.

Purpose: UI-06 (chat with SSE streaming) appears wired but is functionally broken — the coach never sees what the user typed. This makes the chat feature unusable.

Output: `chat_stream` accepts a `message` query param, appends it to the loaded history before streaming, and persists both the user message and the assistant reply via `save_messages` (reusing the assistant_sink mechanism added to sse_generator in 04-12).
</objective>
