<objective>
Close UAT GAP 1 (Test 5: "redirects to /login after signing in"). After a user clicks the magic link, the app lands on /auth/callback where AuthCallbackScreen performs the PKCE code exchange. The race: useAuth's getSession() resolves with a null session BEFORE the exchange completes and writes {session: null, isLoading: false} into the auth store. AuthGate then sees a non-loading, null session and immediately redirects to /login, aborting the callback flow.

The 04-14 fix added getSession() seeding and a transient-null guard on onAuthStateChange, but it did NOT prevent the initial getSession() seed from poisoning the store while on /auth/callback. This plan adds that single guard.

Purpose: UI-01 (auth + navigation shell). A user who completes the magic-link flow must land in the app, not be bounced back to /login.

Output: useAuth.ts skips the {isLoading: false, session: null} seed write when the current path is /auth/callback, letting AuthCallbackScreen own session population. All other routes behave exactly as before.
</objective>

    In useAuth.ts, inside the getSession().then() callback (lines 18-25), add a guard so the seed write is skipped when the current path is the auth callback route.

    Detect the callback route from window.location.pathname. Treat the path as a callback when it includes the substring "/auth/callback" (covers /auth/callback and any query/hash suffix). Compute this once at the top of the effect (e.g. a const onAuthCallback derived from window.location.pathname) so both the seed callback and any later logic read a stable value.

    In the getSession().then() callback: keep the existing `if (!active) return` guard, then add — when onAuthCallback is true AND the resolved session is null, return early WITHOUT calling setAuth. Rationale: on /auth/callback a null session means the PKCE exchange has not completed yet; AuthCallbackScreen will populate the session and onAuthStateChange will fire SIGNED_IN. Writing isLoading:false + session:null here is exactly the poison that triggers the /login bounce.

    Edge cases to preserve:
    - When onAuthCallback is true and getSession() resolves with a NON-null session (e.g. detectSessionInUrl already completed before this callback runs), DO call setAuth normally — a real session is never withheld.
    - When onAuthCallback is false (every other route), behavior is unchanged: setAuth is always called, including with a null session, so AuthGate correctly redirects unauthenticated users to /login.
    - Do NOT change the onAuthStateChange handler (lines 30-39); its SIGNED_OUT-only clearing guard stays as-is.

    Add a brief code comment on the new guard explaining that AuthCallbackScreen owns session population on /auth/callback, so the initial null seed must not run there. Do not introduce react-router hooks (useLocation) inside this effect — useAuth may be mounted above the Router; read window.location.pathname directly as the rest of the file's pattern allows.
