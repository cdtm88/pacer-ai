<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Supabase Auth | Magic-link flow; access_token (JWT) is the credential for all API calls |
| browser → FastAPI | Every apiFetch carries the JWT; SSE carries it as ?token= |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-10 | Spoofing | LoginScreen magic link | mitigate | Auth handled by Supabase signInWithOtp; no password handling in app code; session token issued only after email link click |
| T-04-11 | Information Disclosure | access_token in client memory | accept | Token lives in the Supabase client session (in-memory/localStorage managed by @supabase/supabase-js); never logged; Google tokens are never stored client-side (handled server-side in 04-07) |
| T-04-12 | Elevation of Privilege | AuthGate bypass | mitigate | Server-side JWT verification (04-02) is the real gate; client AuthGate is UX only and cannot be relied on for security |
| T-04-13 | Spoofing | ?token= in SSE URL | accept | Documented tradeoff (Open Question 1): short-lived token in query string for EventSource; refreshed by supabase-js; logs suppress token in deployment |
</threat_model>
