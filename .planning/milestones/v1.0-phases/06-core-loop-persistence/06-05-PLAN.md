<objective>
Make ride data flow through the persisted plan correctly and survive Vercel serverless. Five defects: (1) the estimated FTP is silently discarded because get_user_ftp reads the wrong dict key (`ftp_watts`, but estimate_ftp_from_rides returns `ftp`), so users stay at the 150W placeholder forever; (2) duplicate FIT re-uploads create a second rides row (no content-hash dedup); (3) the ride-session link is never written and matches "first session today" ignoring date/status, so sessions are never marked completed and later counted missed; (4) the pipeline uses FastAPI BackgroundTasks, which Vercel freezes after the response so the work may never run; (5) the old single-step PMC block is wrong and must be replaced by the recompute-from-scratch call. Fix all five. Reuse the content-hash dedup UNIQUE constraint and profiles.ftp column from 06-01 and recompute_pmc_for_user from 06-03.

Phase 6 has no new requirement IDs; FIT-04/FIT-05/TOOL-03 are the existing requirements this plan makes actually functional.

Purpose: Real ride data must update FTP, PMC, and session completion idempotently, inline within the request.
Output: fixed get_user_ftp with write-back, content-hash dedup, ride-session link, inline-awaited pipeline calling recompute_pmc_for_user.
Locked deploy constraint (2026-07-03): Vercel serverless only; the pipeline must be inline-awaited, never deepen BackgroundTasks. Orchestrator decision (OQ2): PMC recompute is triggered only from ride upload.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-06 | Tampering | content-hash dedup race | high | mitigate | Pre-check SELECT is user-scoped; the authoritative guard is the DB `UNIQUE (user_id, content_hash)` constraint (06-01) whose violation is caught and returns duplicate=true, closing the two-concurrent-uploads race |
| T-06-07 | Spoofing | ride ownership | high | mitigate | user_id is sourced only from the verified JWT sub (existing pattern, unchanged); all ride/session/profile writes are user-scoped |
| T-06-08 | Tampering | profiles.ftp write-back | medium | mitigate | The write-back update is filtered by `.eq("user_id", user_id)`; a ride can only affect its own user's FTP |
| T-06-12 | Tampering | content-addressed storage path | low | mitigate | Hash-based Storage path removes client-filename overwrite/collision surprises (side benefit of dedup) |
| T-06-SC | Tampering | package installs | high | accept | No new packages introduced |
