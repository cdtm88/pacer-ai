<objective>
Make adaptation checks idempotent and give macro replans a working two-phase confirm flow. Today every `POST /adaptations/check` re-applies the same 20% cuts and +1-day shifts and writes a new log row because consumed signals are never marked (missed sessions stay `status='planned'`); `POST /adaptations/sessions/{id}/missed` self-defeats because it sets `status='missed'` while `detect_signals` only scans `status='planned'`; the 30% shift guard is mathematically dead (the macro generator moves every session exactly +1 day while the guard only counts moves greater than 1 day); and `apply_macro_replan`'s `needs_confirmation` branch returns a summary that nothing can ever apply (D-19 dead end). Fix all four with `trigger_session_ids` bookkeeping (uniform across micro and macro per orchestrator decision), a status flip on missed, a real shift in the macro generator so the guard fires, and a new `POST /adaptations/{id}/confirm` endpoint that applies the stored proposal. Also fix the compliance TypeError when tss is None.

Phase 6 has no new requirement IDs; ADAPT-01/02/03/04 and TRANSP-02 are the existing requirements this plan makes actually correct.

Purpose: Adaptations must consume each signal exactly once and macro replans must be confirmable.
Output: idempotent detect_signals, trigger_session_ids + status flip, working shift guard, confirm endpoint, compliance None-guard.
Orchestrator decisions honored: proposed macro replans auto-supersede on the next /check (OQ1: yes); apply_micro_adjustment gets trigger_session_ids bookkeeping too (OQ3: yes, uniform).
</objective>
