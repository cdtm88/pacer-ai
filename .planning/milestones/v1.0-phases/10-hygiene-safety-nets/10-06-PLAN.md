<objective>
Close the single failing Phase 10 item (item 7: "CI runs pytest+vitest+ruff") by reverting CI back to D-05's originally-locked scope and confirming it green on a REAL GitHub Actions run.

The `e2e` (Playwright) job was added during this phase's own code-review-fix cycle (10-REVIEW-FIX.md, WR-02) — beyond D-05's locked "ruff+pytest+vitest only" scope — and its "self-sufficient in CI" claim was only validated by local YAML parsing, never a live run. On real GitHub Actions it fails reproducibly (2/2 observed runs: locator timeouts, hover interactions that don't fire headless, element-not-found across Login/Agenda/Settings/Session specs). Stabilizing dozens of flaky Playwright specs for headless CI is a large, risky undertaking; reverting one out-of-scope job addition is the faster, lower-risk path. Separately, the `frontend` (vitest) job carries a pre-existing Phase-09 flake (`session.test.tsx` WebStorage race) that also manifested as a real red build.

Purpose: Restore the reliable "green safety net" the phase is named for — the `backend` job is already solid (2/2 real runs green); this plan removes the two independent sources of false-red (the out-of-scope e2e job, and the pre-existing frontend flake) and PROVES the result against a live run, not local checks.

Output: A two-job report-only CI workflow (backend: ruff+pytest; frontend: vitest) that is confirmed green on a real GitHub Actions run, with the e2e exclusion documented as an explicit decision.
</objective>
