# Phase 10 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Scope Note (read first)

Phase 10 is a hygiene/infrastructure phase (stale SSE test fixes, contract tests, SSE token exchange, rate limiting, CI, repo cleanup — see `10-CONTEXT.md`). Seven of the eight scope items have **zero user-visible surface**: backend auth plumbing, Python/Playwright test fixes, CI config, and repo cleanup produce no rendered UI change at all.

The **one** plausible user-facing surface is item 6: rate limiting on `chat/stream` and `onboarding/start`. When a user (or a runaway retry loop) hits the limit, the UI must show *something* instead of a silently dead stream or a generic "connection failed" message that invites a pointless immediate retry.

This spec does **not** introduce new screens, new components, or a visual redesign. It extends the existing Phase 9 `StreamErrorBanner` error-copy contract with one new terminal-error variant, and locks one interaction rule (skip silent auto-retry when the error is a rate limit, since retrying immediately against a rate limit is self-defeating). Everything else below documents the *existing, unchanged* design system for reference so the checker/executor can confirm no drift — not because this phase changes it.

## Copywriting Contract

Only new copy this phase: the rate-limit message shown in the existing `StreamErrorBanner`. No new CTA, no new empty state, no new destructive action.

| Element | Copy |
|---------|------|
| Rate-limit error — chat (`chat/stream`) | "You're sending messages a bit fast. Wait a moment and try again." — rendered via `<StreamErrorBanner variant="chat">`, existing "Retry" button relabeled copy unchanged |
| Rate-limit error — onboarding (`onboarding/start`) | Same copy, rendered via `<StreamErrorBanner variant="onboarding">` (matches the existing onboarding terminal-error placement in `OnboardingScreen.tsx`) |
| Existing generic connection-failure copy (unchanged, for contrast) | "Stream error" / "Connection failed." / "Connection lost. Please try again." (`useSSEStream.ts`, `OnboardingScreen.tsx`) — rate-limit copy must NOT reuse these generic strings, so the user understands *why* the stream stopped and that immediate retry won't help |
| Primary CTA | N/A — no new CTA. Reuses the existing "Retry" button (icon + label unchanged) |
| Empty state | N/A — not introduced this phase |
| Destructive confirmation | N/A — not introduced this phase |

**Interaction rule (locks planner/executor behavior — not just copy):** when the backend signals a rate-limit condition (e.g. an SSE `error` event with `code: "rate_limited"`, or an `onboarding/start` HTTP 429), the frontend must **skip the existing silent auto-retry-with-backoff** (`MAX_RETRIES`/`BACKOFF_MS` in `useSSEStream.ts` and `OnboardingScreen.tsx`) and go straight to the terminal `StreamErrorBanner` with the rate-limit copy above. Auto-retrying immediately against a rate limit compounds the problem it's meant to prevent. This is the one interaction decision this phase adds; it does not change the retry policy for any other error type (auth, connection, trust-violation, save failure all keep the existing 2x backoff-then-banner behavior).

Backend note for the planner (not a UI concern but affects whether the above is reachable): to let the frontend distinguish "rate limited" from "generic failure" without inventing new error-handling paths, the rate-limit response should fit the *existing* error shapes already consumed by the frontend — either an SSE `error` frame `{"code": "rate_limited", "message": "..."}` (matching `backend/routes/chat.py`'s existing error-frame schema) for `chat/stream`, or a JSON body on the 429 response for `onboarding/start` that the existing `!res.ok` branch can read a `detail`/`message` string from. No new frontend error-handling architecture is required — only a code/message check that special-cases skipping retry.
