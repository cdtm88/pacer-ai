<objective>
Add a best-effort, in-process rate limiter to the two LLM-backed endpoints (`chat/stream`, `onboarding/start`) so a runaway client retry loop cannot burn unbounded Anthropic spend (D-02, D-03). Hand-rolled per RESEARCH — a ~30-line token-bucket dependency keyed by `user_id`, chained after `Depends(get_current_user)`. `slowapi` is deliberately NOT used (flagged `[SUS]` by the legitimacy gate; its IP-keyed decorator model doesn't fit this app's `Depends(get_current_user)` pattern). The frontend renders the existing `StreamErrorBanner` with rate-limit copy and skips the silent auto-retry (10-UI-SPEC interaction rule).

Purpose: Scope item 6 — a cost/abuse safety net, explicitly non-distributed (accepted per D-02 for a single-user app; Fluid Compute may under-count across instances).

Output: `backend/rate_limit.py`, both endpoints wired, and both frontend error paths special-casing rate limits.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → LLM endpoints | a runaway/abusive client can drive unbounded Anthropic API spend |
| rate-limit key selection | keying by IP instead of identity would let Vercel edge-shared IPs bypass or over-limit |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-04-01 | Denial of Service (cost) | POST /onboarding/start, GET /chat/stream | high | mitigate | In-process sliding-window token bucket (~10 req/60s) keyed by user_id caps runaway retry-loop spend (D-02, D-03); Task 2 proves the Nth+1 request is limited |
| T-10-04-02 | Spoofing / bypass | rate-limit key dimension | medium | mitigate | Key by post-auth `user_id`, never by `Request`/IP (Vercel edge IPs are shared); Task 1 asserts no `get_remote_address`/`request.client` usage |
| T-10-04-03 | Denial of Service (under-count) | Fluid Compute multi-instance | low | accept | In-process limiter can under-count across cold starts / warm instances; accepted best-effort for a single-user app per D-02 (Redis/Upstash deferred) |
| T-10-04-04 | Denial of Service (self-inflicted retry storm) | frontend auto-retry on rate limit | medium | mitigate | Frontend skips the silent auto-retry when the error is `rate_limited` (Task 3), so the client cannot compound the limit it just hit |
</threat_model>
