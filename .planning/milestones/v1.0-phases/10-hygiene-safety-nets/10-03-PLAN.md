<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → GET /chat/stream ?token= | EventSource cannot send headers; a credential rides in the query string and lands in access logs |
| POST /chat/token minting | server-side signing of a bearer-equivalent credential |
| sse_token ↔ real Supabase JWT | two token types verified by the same dependency must never be confused |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-10-03-01 | Information Disclosure | `GET /chat/stream?token=` query param | high | mitigate | Replace the full Supabase JWT in the SSE URL with a short-lived (~60s) namespaced `sse_token`; the leaked value can never be the real Supabase JWT (Task 3) |
| T-10-03-02 | Spoofing (token confusion) | `get_current_user` verify path | high | mitigate | `typ: "sse_token"` claim namespace guard + dedicated `SSE_TOKEN_SECRET` (never `SUPABASE_JWT_SECRET`); Task 1 test 4 asserts a real Supabase JWT is not accepted as an sse_token and vice versa |
| T-10-03-03 | Elevation (weak secret) | `SSE_TOKEN_SECRET` | high | mitigate | High-entropy secret (`openssl rand -hex 32`); HS256 via PyJWT, never hand-rolled crypto; missing secret → HTTP 500, never an unsigned token |
| T-10-03-04 | Spoofing (token replay) | ephemeral sse_token | medium | accept | 60s `exp` bounds the replay window; no nonce/single-use tracking since that needs durable state, explicitly ruled out by D-04. Accepted for a single-user app |
| T-10-03-05 | Information Disclosure | `.env.example` / committed secrets | low | mitigate | Only a commented empty placeholder is added to `.env.example`; the real value lives in Vercel env (user_setup), never committed |
</threat_model>
