<objective>
Apply the same retry-then-terminal-error recovery to onboarding's fetch-based streams (item 13, D-05).

Onboarding uses POST fetch + manual ReadableStream SSE parsing (NOT EventSource) because it needs an Authorization header and JSON body — so it cannot literally share useSSEStream (Pitfall 1). What it shares is the retry POLICY and the StreamErrorBanner COMPONENT (created in 09-01). Today: runStream's failure paths clear isStreaming but offer no Retry, and handleConfirm's !res.ok branch has NO setStreamError at all and silently falls through to pollForProfile — the "spinner sticks forever" bug D-05 describes for the confirm step.

Purpose: Onboarding recovers from stream faults with the same UX as chat, and the confirm step never sticks the spinner.
Output: mirrored retry loops in runStream + handleConfirm; StreamErrorBanner rendered in place of the static banner; updated onboarding tests.
</objective>
