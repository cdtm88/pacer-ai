Add the Phase 4 DB migration and the six backend read/create endpoints the frontend cannot function without. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → FastAPI | Untrusted HTTP requests cross here; user_id must come from a verified JWT, never request input |
| FastAPI → Supabase | Service-role key bypasses RLS; application-layer user_id filter is the access-control gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Elevation of Privilege | new read endpoints in sessions.py/rides.py | mitigate | All handlers use Depends(get_current_user) from api.auth (04-02); user_id is taken from the JWT sub claim, never from query/body |
| T-04-03 | Information Disclosure | GET /profiles/me, /sessions/* | mitigate | Supabase queries filter `.eq("user_id", user_id)` from the JWT so cross-user reads return no rows |
</threat_model>
