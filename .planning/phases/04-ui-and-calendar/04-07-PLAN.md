<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Google OAuth | Redirect-based consent; the authorization code crosses back to our callback |
| our callback → Google token endpoint | Code-for-token exchange happens server-side only |
| FastAPI → users.google_tokens | Tokens are encrypted at rest; the encryption key is server-only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-21 | Spoofing (CSRF) | /calendar/callback | mitigate | OAuth state parameter stored server-side in Supabase and verified at callback before token exchange |
| T-04-22 | Information Disclosure | google_tokens at rest | mitigate | Fernet (authenticated AES) encryption; tokens never stored plaintext, never sent to the browser, never logged (CAL-03) |
| T-04-23 | Denial of Service | calendar sync in request path | mitigate | Sync is fire-and-forget background work; failures swallowed so the adaptation/chat path cannot 500 (CAL-04) |
| T-04-24 | Elevation of Privilege | OAuth scope | mitigate | Request only calendar.events (least privilege), not full calendar access |
| T-04-25 | Tampering | google-api-python-client / cryptography install | mitigate | Blocking human-verify legitimacy checkpoint (Task 0) confirms [ASSUMED] packages on PyPI before install |
| T-04-SC | Tampering | pip installs | mitigate | slopcheck + the blocking legitimacy checkpoint; never auto-approved |
</threat_model>
