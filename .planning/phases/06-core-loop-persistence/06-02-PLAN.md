<objective>
Make a generated plan become real database state. Today `generate_plan` returns sessions with `plan_id: None` and nothing is ever inserted into `plans` or `sessions` (grep-verified), so Today/Agenda/ZWO/calendar/adaptations all read empty tables and the core product loop is broken end-to-end. Add a persistence hook in `dispatch_tool` (the orchestration layer that already special-cases `generate_plan` for user_id injection) that inserts a `plans` row and the resolved `sessions` rows, then rewrites `result.value["plan_id"]` before the tool result returns to Claude. Keep `sports_science/plan.py` pure and untouched (locked invariant). Also write `profiles.lthr` from `lthr_estimate` in `save_profile` (schema column added in 06-01).

Phase 6 has no new requirement IDs; PLAN-01/PLAN-04/ONBD-04 are the existing requirements this plan makes actually functional (a confirmed plan becomes persisted per-session state).

Purpose: A confirmed plan must exist as rows the rest of the app reads.
Output: `_persist_generated_plan`, `_resolve_scheduled_date`, a `generate_plan` branch in `dispatch_tool`, and lthr write-back in `save_profile`.
</objective>
