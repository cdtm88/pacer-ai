    1. ALTER TABLE public.profiles ADD COLUMN back_status text NOT NULL DEFAULT 'none' CHECK (back_status IN ('none','mild','moderate')), ADD COLUMN weekly_hours numeric, ADD COLUMN preferred_days text[], ADD COLUMN rpe_baseline text, ADD COLUMN lthr_estimate numeric.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend → Supabase Postgres | Migration DDL crosses into the database; new tables must enforce per-user isolation |
| authenticated user → tables (read) | Users reading their own plans/adaptations via RLS-protected SELECT |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | plans, adaptations tables | mitigate | RLS enabled with `USING (user_id = auth.uid())` on both new tables; user cannot read another user's plan or adaptation log |
| T-03-02 | Elevation of Privilege | RLS policies | mitigate | No INSERT/UPDATE policies granted to anon/authenticated roles; backend writes only via SERVICE_ROLE_KEY (TRUST architecture: backend is the sole writer) |
| T-03-03 | Tampering | FK constraints | mitigate | sessions.plan_id and rides.session_id use REFERENCES with ON DELETE behavior; orphaned rows prevented at DB level |
| T-03-SC | Tampering | supabase CLI invocation | accept | No new package installs in this plan; `supabase db push` is an existing trusted toolchain step from Phase 1 |
</threat_model>
