# Phase 11 — UI Design Contract

> Visual and interaction contract for the Ride Analysis Dashboard. Generated by gsd-ui-researcher from `11-CONTEXT.md` (locked decisions D-11-01..D-11-07) and `11-RESEARCH.md` (codebase-verified patterns). Verified by gsd-ui-checker.

**Design-system anchor:** This phase ships against the **current shipped design system** (Inter, `--color-*` tokens in `index.css`, `card-elev`, `ZoneChip`, `PmcChart`, `BottomTabBar`/`DesktopSidebar`, existing spinner pattern) — not the Phase 12 "Athletic Redesign" target (dark cockpit, Barlow Condensed display face, filled-pill nav, `--cockpit-*` tokens), which has an approved UI-SPEC but has not executed. Phase 12's own roadmap entry explicitly defers to this phase: *"if Phase 11 lands first, its Analysis screen adopts this spec [12]"* — i.e. any redesign of the Analysis screen to the athletic system is Phase 12's job when it runs, not this phase's. Building against Phase 12's not-yet-real tokens (`--cockpit-*`, `.stat-num-hero`, Barlow Condensed) would be visually inconsistent with every other screen in the app today and is explicitly out of scope here.

## Spacing Scale

Confirmed 8-point scale, matches the rest of the codebase (no phase-specific exceptions to the scale itself):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, zone-distribution bar segment gaps |
| sm | 8px | Chip/badge padding, readout-row item gaps |
| md | 16px | Chart card padding (`card-elev`, matches `PmcChart` exactly: `16px 12px 8px`), default element spacing |
| lg | 24px | Section spacing between chart stack and time-in-zone section |
| xl | 32px | — (not used this phase) |
| 2xl | 48px | — (not used this phase) |
| 3xl | 64px | — (not used this phase) |

Exceptions:
- Touch targets: 44×44px minimum. Nav tabs going from 4→5 (Pitfall 6, 11-RESEARCH.md) shrinks each tab's width but the 56px bar height keeps every tab ≥44px tall — no redesign needed, just a visual smoke-check at implementation time.
- Chart canvas height: 200px per `RideChart` panel (not a spacing-scale value — deliberately shorter than `PmcChart`'s 240px because up to 5 channel charts stack vertically on one screen; see Screen & Component Contract).

## Typography

Capped at 3 distinct sizes, 2 weights (Inter 400/600 — consistent with `PmcChart`'s and `StatTile`'s existing "chart chrome" convention, and forward-compatible with Phase 12's stated weight budget without depending on it):

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Micro (axis) | 11px | 400 | 1.3 | Recharts X/Y axis ticks — exact `AXIS_TICK` reuse from `PmcChart.tsx`: `{ fontSize: 11, fill: 'var(--color-ink-3)', fontVariantNumeric: 'tabular-nums' }` |
| Body | 13px | 400 | 1.4 | Synced readout-row values, chart legend text, time-in-zone row seconds/pct, empty/error state body copy |
| Label | 13px | 600 | 1.4 | Chart card titles ("Power", "Heart rate", "Cadence", "Speed", "Elevation"), lap chip text, time-in-zone row zone names |
| Heading | 18px | 600 | 1.3 | "Time in zone" section heading, empty/error state heading |

Screen title ("Analysis") is **inherited unchanged** from `AppLayout.tsx`'s existing `<header><h1>` (20px/700, `--color-ink`) — this phase adds a `ROUTE_TITLES` entry, it does not modify the header component or its type scale.

Weight budget: 2 weights only (400 regular, 600 semibold) across every new component this phase introduces. No 500 or 700 in new code, even though both exist in the loaded font set from older screens.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--color-bg` `#F2F4F7` | App background (inherited, `AppLayout`/`body`) |
| Secondary (30%) | `--color-surface` `#FFFFFF` (chart cards via `card-elev`) + `--color-ink`/`--color-ink-2` (text) + `--color-line` (borders, dividers, lap reference lines) | Chart card surfaces, readout-row bar, section dividers |
| Accent (10%) | `--color-brand` `#1F6FE5` | Analysis nav tab active state (existing `BottomTabBar`/`DesktopSidebar` pattern, unchanged), "View analysis" link text on each `RideRow`, focus rings |
| Destructive | `--color-bad` `#C0341D` | Not used this phase — no destructive actions exist in Ride Analysis. Declared for template completeness only. |

Accent reserved for: the Analysis tab's active nav state (icon + label + indicator, same mechanism as the other 4 tabs), the new "View analysis" text link on `RideRow`, and focus-visible rings on the readout row / lap chips. Never used for a chart line stroke or a decorative fill — those are the channel-color and zone-color systems below, which are a separate data-encoding layer outside the 60/30/10 budget (same treatment Phase 12's spec gives zone colors).

### Channel line colors (new — one prescriptive token per channel, not part of the 10% accent budget)

| Channel | Token | Hex | Rationale |
|---------|-------|-----|-----------|
| Power | `--color-zone-endurance` | `#228BE6` | Matches `11-RESEARCH.md` Pattern 3's verbatim code example |
| Heart rate | `--color-warm` | `#FF8A5C` | Deliberately distinct from the `--color-zone-*` set used one section below in the time-in-zone bar — keeps "this is the raw HR trace" visually separate from "this is a zone classification" |
| Cadence | `--color-ink-3` | `#888C93` | Neutral/secondary channel, de-emphasized relative to power/HR |
| Speed | `--color-blue-4` | `#53A5EC` | Distinct lighter blue, does not collide with power's endurance-zone blue or `--color-brand` |
| Altitude | `--color-good` | `#2B8A5B` | Conventional elevation/terrain green |
| Lap `ReferenceLine`s | `--color-line` | `#DFE0E2` | Dashed (`strokeDasharray="3 3"`), matches `PmcChart`'s `ReferenceLine` treatment exactly |

`distance` (RIDE-01 extracts it, but per 11-RESEARCH.md Open Question 1) is **readout-only** — it appears in the synced readout row but does not get its own chart or channel color.

### Time-in-zone colors (reused exactly, no new hex — Pattern 4 / Don't Hand-Roll)

| Zone | Token | Hex |
|------|-------|-----|
| Recovery | `--color-zone-recovery` | `#2B8A5B` |
| Endurance | `--color-zone-endurance` | `#228BE6` |
| Tempo | `--color-zone-tempo` | `#F0A030` |
| Threshold | `--color-zone-threshold` | `#E8590C` |
| VO2max | `--color-zone-vo2` | `#C92A2A` |

Reuse `ZoneChip.tsx`'s `ZONE_VAR`/`ZONE_LABEL` maps directly for the time-in-zone rows (badge treatment identical to every other zone display in the app: `color-mix(in srgb, var(--color-zone-X) 15%, transparent)` background, zone-color text). Do not hand-roll a second zone-color map (11-RESEARCH.md Don't Hand-Roll table, Pattern 4).

Contrast rules (inherited, unchanged, binding): `#228BE6` is 3.56:1 on white — large text/fills/lines only, never small body text in that exact hex; small blue text uses `--color-blue-7` `#1B73C0` if ever needed (not expected this phase, no small blue text is introduced).

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary "CTA" (RideRow → Analysis link, RIDE-11) | **"View analysis"** — small brand-colored text link, right-aligned in each `RideRow`, added alongside (not replacing) the existing tap-to-expand compliance-summary interaction. A separate, explicit link avoids ambiguity between "expand this row" and "navigate to the Analysis screen," and avoids nesting two interactive elements inside one `<button>` (the row's existing expand toggle). |
| Empty state heading (Analysis, zero rides) | **"No rides yet"** — verbatim match to `ProgressScreen`'s existing empty state heading (11-RESEARCH.md Open Question 2 recommendation: reuse the established pattern, don't invent new copy) |
| Empty state body (Analysis, zero rides) | **"Upload a .FIT file to see your ride analysis."** — adapted from `ProgressScreen`'s "Upload a .FIT file from Zwift or your head unit to see your history." (same pattern, same component shape, reworded for this screen's context) |
| Error state — generic stream fetch failure | **"Could not load ride data. Tap to retry."** — mirrors `ProgressScreen`'s existing "Could not load history. Tap to retry." exactly, retitled for ride-stream data; same tappable-text-button treatment (`--color-bad`, 14px, centered) |
| Error state — corrupt file (backend 422, T-11-03) | **"Could not read this ride file. It may be damaged or in an unsupported format."** — no retry action offered; retrying cannot fix a corrupt source file, so this state omits the "Tap to retry" affordance the generic error uses |
| Error state — ride not found (backend 404) | **"This ride couldn't be found."** + a text link **"Back to ride log"** returning to `/progress` |
| Destructive confirmation | Not applicable — this phase introduces no destructive actions (no delete, no irreversible state change) |

No em dashes in any of the above (verified — commas and separate sentences used throughout, per project constraint).

## Screen & Component Contract

### `RideChart.tsx` (new) — RIDE-07, RIDE-08, RIDE-09

- One `card-elev` panel per **present** channel (power, heart_rate, cadence, speed, altitude — `distance` excluded, readout-only). Absent channels (D-11-03) render **no card at all** — not a disabled/greyed placeholder, not an "unavailable" message. This is a deliberate non-visual decision: an indoor Zwift ride with no elevation shows 4 chart cards, not 5 with one empty.
- Panel structure, copied from `PmcChart.tsx` / 11-RESEARCH.md Pattern 3 exactly:
  - `card-elev` wrapper, padding `16px 12px 8px`.
  - Card title (Label role: 13px/600) top-left, e.g. "Power", "Heart rate".
  - Chart canvas: `height: 200px` (not `PmcChart`'s 240px — shorter because multiple panels stack on one screen).
  - `<ResponsiveContainer>` → `<LineChart syncId="ride">` (D-11-06 — shared `syncId` across every panel, this is the entire cross-chart hover-sync mechanism, no custom code).
  - `<CartesianGrid stroke="var(--color-line-2)" vertical={false} />`
  - X axis: seconds-since-start data, ticks formatted as `"Nm"` (integer minutes: "5m", "10m", "15m" — not "5:00"). Micro role (11px/400, `--color-ink-3`, tabular-nums).
  - Y axis: Micro role, no axis line, `width: 38`.
  - `<Line>` stroke = the channel's color token from the table above, `strokeWidth: 1.5`, `dot={false}`, `isAnimationActive={false}` (matches every existing Recharts line in this codebase).
  - `<ReferenceLine>` per lap boundary (from `stream.laps`), `stroke="var(--color-line)"`, `strokeDasharray="3 3"`. **Do not pass `isFront` or `alwaysShow`** — both removed in Recharts v3 (11-RESEARCH.md Anti-Pattern, confirmed against the installed `^3.8.1`).
- **Synced readout row** (RIDE-08): one `card-elev` bar positioned directly above the chart stack (not sticky — a fixed/sticky bar competing with `AppLayout`'s own sticky header inside a scrolling `<main>` adds complexity for a marginal gain; render it as the first element in the scroll flow instead). Contents, left to right, wrapping on narrow viewports:
  - Lap chip: small pill, `background: color-mix(in srgb, var(--color-brand) 10%, transparent)`, text "Lap N" (Label role, 13px/600, `--color-brand`), `border-radius: 999px`, padding `2px 10px`.
  - Time: formatted **`"Mm SSs"`** (e.g. "5m 32s", "0m 45s") — this is a **new formatter**, deliberately distinct from `DuringSessionScreen`'s existing `MM:SS` colon-format timer (11-RESEARCH.md: not a reuse case, RIDE-08 specifies the letter format explicitly). Body role, `stat-num` class (tabular-nums), `--color-ink`.
  - One value per **present** channel at the hovered point, each as a small label/value pair: Micro-role unit label above (e.g. "POWER", "HR", "CAD", "SPEED", "ELEV", "DIST" — uppercase, `--color-ink-3`, matches `StatTile`'s label treatment) + Body-role value below (`stat-num`, `--color-ink`, channel's color token used only for the small unit label, not the value text, to keep the readout row legible against 5+ colors at once).
  - Before any point is hovered: show the values at `t=0` (first record) as the default/rest state, not blank cells — avoids an empty-looking bar on first paint.
- **Time-in-zone section** (RIDE-09): rendered **only when** `stream.hr_zone_distribution` is not `null` (covers both "no heart_rate channel present" and "user has no LTHR / Branch C," per 11-RESEARCH.md Pitfall 3 — both cases omit this section entirely, no disabled/empty variant).
  - Section heading: "Time in zone" (Heading role, 18px/600), `margin-bottom: 16px` (`md` spacing), positioned after the chart stack (`lg` = 24px gap above it).
  - A single horizontal stacked bar: 5 segments in zone order (recovery → vo2), each segment width = that zone's `pct`, fill = that zone's `--color-zone-*` token, bar `height: 12px`, `border-radius: 6px`, segments touching (no gap) with the bar's own `border-radius` clipping the ends.
  - Below the bar: one row per zone using `ZoneChip` (Pattern 4 reuse — zone label + color badge, unchanged component) + seconds formatted as `"Mm SSs"` (same formatter as the readout row) + percentage (`"41%"`), both Body role, `stat-num`, right-aligned in the row. Zones with `0` seconds still render (don't hide — the shape of the zone list is informative even at zero).

### `AnalysisScreen.tsx` (new) — RIDE-10

- Route `/rides/:rideId` and default `/analysis` (falls back to the latest ride via the already-fetched `getRides()` list — no new backend endpoint, per 11-RESEARCH.md Don't Hand-Roll).
- Loading state: reuse the **exact existing spinner pattern** from `AuthGate`/`FirstRunGate` (`w-8 h-8 rounded-full border-2 border-t-transparent animate-spin`, `border-color: var(--color-blue-6)`), centered in the scroll area. This is the established loading affordance used twice already in this codebase — do not introduce a new shimmer/skeleton pattern for a third, inconsistent loading treatment.
- Empty state (zero rides for this user): centered text block, matches `ProgressScreen`'s empty-state layout exactly (`textAlign: center`, `paddingTop: 24`, Heading role for the "No rides yet" line, Body role for the body copy below it) — see Copywriting Contract above for exact copy.
- Error states: see Copywriting Contract above (generic / corrupt-file / not-found are three distinct states, not one generic error).
- Header title: add a `ROUTE_TITLES` entry `'/analysis': 'Analysis'` in `AppLayout.tsx`. **Implementation note:** `ROUTE_TITLES` is keyed by exact static pathname and will not match the dynamic `/rides/:rideId` segment — the executor needs a small fallback (e.g. `pathname.startsWith('/rides/') ? 'Analysis' : ROUTE_TITLES[pathname] ?? 'PacerAI'`) so both routes show "Analysis" in the header, not the raw path or the default "PacerAI" fallback.

### Nav tab (new) — RIDE-10, 11-RESEARCH.md Pitfall 6

- Icon: `Activity` (lucide-react), Label: "Analysis", route `/analysis`.
- Order in both `BottomTabBar.tsx` and `DesktopSidebar.tsx`: **Today, Agenda, Progress, Analysis, Coach** — groups the three data/review destinations (Agenda, Progress, Analysis) together ahead of Coach chat, which stays last as the existing convention.
- Visual treatment: identical to the existing 4 tabs, no new styling — active state uses the existing dot indicator (`BottomTabBar`) / left-border + tinted background (`DesktopSidebar`), `--color-brand`. Do not adopt Phase 12's filled-pill nav treatment; that lands only when Phase 12 executes.
- Going from 4→5 tabs shrinks each mobile tab's width proportionally (Pitfall 6) — not a blocker, implement per D-11-05, but flag a manual visual smoke-check (existing `AppLayout.test.tsx` / a screenshot) as part of Wave 2 verification.

### `RideRow.tsx` (modified) — RIDE-11

- Add the "View analysis" text link (Copywriting Contract above) as a new element in the collapsed row, alongside the existing date / compliance chip / stats layout — do not remove or restyle any existing element in this file as part of this phase (the `<table>`→bars redesign described in Phase 12's UI-SPEC is that phase's job, not this one's).
