# Phase 09 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase Nature

This is bug-fix/resilience work on **existing** screens (ChatScreen, OnboardingScreen, router/AppLayout, FitUploadZone) per `09-CONTEXT.md`. There is no new visual design system to establish — the app's tokens (`frontend/src/index.css`), fonts, and component patterns already exist and are production-proven. This contract does **not** restate the whole design system; it declares the visual/interaction rules for the five specific new UI surfaces this phase introduces, and pins them to the existing token set so nothing new is invented.

Surfaces in scope:
1. Chat SSE error banner + manual Retry button (D-02)
2. Onboarding confirm-stream error state, mirrors #1 (D-05)
3. Router error boundary fallback, per-route inside `AppLayout` (D-09, D-10)
4. Stale-session silent-discard — explicitly **no new UI** (D-06)
5. Upload progress indicator + drag-drop `.fit` validation error (Claude's Discretion item)

## Design System (existing — referenced, not redefined)

| Property | Value |
|----------|-------|
| Tool | shadcn/ui (already initialized — `frontend/components.json`) |
| Preset | style: new-york, baseColor: neutral, cssVariables: true |
| Component library | Radix primitives via shadcn (`components/ui/`) |
| Icon library | lucide-react |
| Font | Inter (`--font-family-sans`, `frontend/src/index.css:42`) |
| Mode | Light mode only, no pure blacks (UI-10 — unchanged) |
| Copy rule | No em dashes in any copy (project-wide CLAUDE.md constraint) |

**Known constraint carried into this phase:** shadcn's `Button` primitive (`components/ui/button.tsx`) references `bg-primary`/`text-primary-foreground`/`bg-destructive` Tailwind utilities, but the project's `@theme` block in `index.css` never defines `--color-primary` / `--color-destructive` / `--background` — only the custom `--color-*` scale (blue, ink, warm, etc.). `variant="default"` and `variant="destructive"` on `Button` are therefore unreliable (no matching CSS generated). This is why high-traffic hand-built surfaces (`ChatInput`, `ChatBubble`, `OnboardingScreen` banners, `FitUploadZone`) all use inline `style={{ ... var(--color-*) }}` instead of `Button` default/destructive variants. **This phase's new elements follow that same proven pattern** — inline styles against the existing `--color-*` custom properties — rather than introducing reliance on the untested `Button` default variant. (Not a fix for the underlying `Button` token gap; that is out of scope for Phase 9 per `09-CONTEXT.md` D-01's 14-item list — flag as a Phase 10 hygiene candidate if not already tracked.)

## Component Specs

### 1. Stream error banner (shared: chat + onboarding confirm-stream)

New shared visual pattern — recommend extracting as one component (e.g. `StreamErrorBanner`) consumed by both `ChatScreen.tsx` and `OnboardingScreen.tsx`'s confirm-stream path, per D-05's instruction to keep the two SSE consumers consistent.

Only rendered **after** the 1-2x silent auto-retry (D-02) has already failed — never on the first transient error (that stays silent, no flash of UI).

Layout: full-width row, flex, `align-items: center`, `justify-content: space-between`, `gap: 8px`.

```
┌──────────────────────────────────────────────────┐
│ ⓘ  {error message}                    [↻ Retry]  │
└──────────────────────────────────────────────────┘
```

Styling (inline style object, matching existing banner precedent in `ChatScreen.tsx:206-223` / `OnboardingScreen.tsx:527-547`):
- `padding: '8px 16px'`
- `backgroundColor: 'var(--color-bg-2)'`
- `borderTop: '1px solid var(--color-bad)'` (chat: banner sits above `ChatInput`) or `border: '1px solid var(--color-bad)', borderRadius: '8px'` (onboarding: banner sits inline in the message column, matching the existing `streamError` block's container treatment)
- Icon: lucide `AlertCircle`, size 16, `color: 'var(--color-bad)'`
- Message text: 13px / 400 / `color: 'var(--color-bad)'`
- Retry button: inline-styled (not shadcn `Button` default — see Design System note above), `display: flex, alignItems: center, gap: 4px`, `padding: '6px 12px'`, `borderRadius: '6px'`, `border: none`, `backgroundColor: 'var(--color-blue-6)'`, `color: 'var(--color-surface)'`, 14px/600, icon lucide `RotateCw` size 14

Interaction:
- Retry re-sends the last user message (chat) or re-invokes the confirm-stream call (onboarding) — does not require the user to retype anything.
- Clicking Retry clears the banner immediately (optimistic) and re-enters the normal streaming state (input disabled, streaming indicator shown) while the retry attempt is in flight.
- If Retry itself fails, the banner reappears (no re-entering silent auto-retry — one manual attempt at a time, matches D-02's "manual Retry button" intent without adding a second hidden retry loop).

### 2. Router error boundary fallback

Per-route boundary nested inside `AppLayout`'s `<Outlet />` (D-10) — nav shell (bottom tab bar / desktop sidebar / header) stays mounted and functional. Fallback fills the content pane only, not the full viewport (do not use `min-h-screen`; use `h-full flex flex-col items-center justify-center` inside `<main>`, consistent with `AppLayout.tsx`'s existing `<main className="flex-1 pb-16 md:pb-0" style={{ minHeight: 0 }}>` container).

```
┌──────────────────────────────────┐
│                                   │
│                                   │
│        Something went wrong      │
│    This page ran into a problem. │
│                                   │
│           [ Reload ]             │
│                                   │
└──────────────────────────────────┘
```

Styling:
- Container: `textAlign: center`, `padding: 24px`, background inherits `var(--color-bg)` (no card/border — matches the plain-text style already used for `FirstRunGate`'s "Could not reach the server" fallback in `router.tsx:104-109`)
- Heading "Something went wrong": 18px / 600 / `color: 'var(--color-ink)'`, `margin: '0 0 8px'`
- Body "This page ran into a problem.": 15px / 400 / `color: 'var(--color-ink-2)'`, `margin: '0 0 16px'`
- Reload button: same inline-styled treatment as Retry (accent blue-6 background, white text, 14px/600), label "Reload", `onClick={() => window.location.reload()}`

Per D-09: no error detail shown, no report/copy-error action, no error ID. This is deliberate — single-user personal app, no support pipeline.

### 3. Stale-session recovery — no UI

Per D-06: on `PersistedSession` id/date mismatch, the app silently discards the stale `localStorage` entry and renders Today's real state. **This UI-SPEC explicitly confirms there is no banner, toast, or dialog to design here.** The checker should verify absence, not presence, of any new element for this behavior. Any implementation that adds a toast/dialog for this case is out of contract.

### 4. Upload progress indicator (`FitUploadZone.tsx`)

Browser upload-progress events are not reliably available via `fetch()` across target browsers (notably iOS Safari, which this app must support per IOS-03) without rewriting `uploadRide` onto `XMLHttpRequest`. Given this is bug-fix/resilience scope, not a feature rewrite, the spec is an **indeterminate** progress indicator, not a byte-accurate percentage.

Add a 3px indeterminate progress bar beneath the existing dropzone content (keep the existing `Loader2` spinner + "Uploading ride..." text unchanged — the bar is additive reinforcement, not a replacement):
- Track: `height: '3px'`, `borderRadius: '2px'`, `backgroundColor: 'var(--color-line-2)'`, `width: 100%`, `marginTop: '8px'`, `overflow: hidden`
- Fill: `backgroundColor: 'var(--color-blue-6)'`, animated indeterminate sweep (CSS `@keyframes` translateX loop, ~1.2s, matches the existing `animate-spin`/`animate-bounce` Tailwind-utility-driven motion style already used in `Loader2`/`ChatBubble`'s `StreamingEllipsis`)
- Only rendered while `isUploading === true`; unmounts on success/error (existing `toast.success`/`toast.error` already handle the terminal states, unchanged)

Drag-drop `.fit` extension validation (currently only validated on the file-picker `accept=".fit"` path, not on `handleDrop`): on invalid extension, do **not** call `uploadRide`; show `toast.error('Only .fit files are supported.')` — reuses the existing `sonner` toast pattern already wired into this component (`toast.error` used for upload failures at line 35).
