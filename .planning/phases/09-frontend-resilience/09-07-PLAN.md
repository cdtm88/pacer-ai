<objective>
Fix conversation-history loss on cache miss (item 4, D-04) — the one item in this phase with a hidden backend sub-task.

Today ChatScreen's useQuery(['active-conversation']) queryFn calls createConversation() unconditionally, so after the 5-minute gcTime evicts the cache, a reload creates a NEW empty conversation row and the prior history disappears. D-04 requires refetching the EXISTING conversation from the DB instead (explicitly NOT a gcTime bump). RESEARCH Open Question 3 flagged that this needs a backend read endpoint — CONFIRMED during planning: no GET /conversations/{id} exists (only POST /conversations/ in conversations_router), but a user-scoped load_conversation(conversation_id, user_id, limit) helper already exists in backend/routes/onboarding.py. So this plan adds a thin GET endpoint wrapping that helper, a frontend fetch function, and the ChatScreen persistence + queryFn branch.

Purpose: Conversation history survives the cache GC window; reloading resumes the same conversation.
Output: GET /conversations/{id}/messages (user-scoped); api.ts getConversationMessages; ChatScreen id-persistence + branch + message hydration; backend + frontend tests.
</objective>

| Boundary | Description |
|----------|-------------|
| client → API (conversation read) | A client-supplied conversation_id crosses into a DB read; it must be validated and scoped to the authenticated principal so one user cannot read another's conversation (ASVS V4 object-level authorization / IDOR) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-07-01 | Information Disclosure / Elevation (IDOR) | GET /conversations/{id}/messages | high | mitigate | Endpoint sources user_id from the verified JWT sub and reuses load_conversation's app-layer ownership filter (WHERE conversation_id AND user_id) plus the existing malformed/foreign-id validation pattern, so a foreign conversation id returns no rows; no new unscoped SELECT is written |
| T-09-07-02 | Tampering | client localStorage conversation id | low | mitigate | A tampered/foreign persisted id cannot leak data because the backend endpoint re-validates ownership server-side; a bad id simply returns no messages, degrading to an empty/new conversation |
| T-09-07-03 | Information Disclosure | error responses | low | mitigate | getConversationMessages surfaces only parsed backend detail strings, not raw internals; the endpoint returns a plain messages list with no other-user metadata |
| T-09-07-SC | Tampering (supply chain) | pip/npm installs | n/a | accept | No new packages introduced |
