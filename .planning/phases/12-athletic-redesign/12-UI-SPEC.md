## Foundation Fixes (Delivery Slice A — do first, everything else depends on it)

### 1. Font loading defect

`frontend/index.html` currently loads only `Inter:wght@400;500;600;700` but `.stat-num` requests weight 800 (synthesized/faux bold — a real defect). Fix by:

- Adding Barlow Condensed to the same Google Fonts link, weights 600 and 700 only.
- Dropping all reliance on Inter 800 — hero numerals move to Barlow Condensed instead (see Typography). Small inline tabular numbers keep Inter but at weight 700 (already loaded).

New link tag:
```
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700&family=Barlow+Condensed:wght@600;700&display=swap" rel="stylesheet" />
```

Add to `@theme` in `index.css`:
```
--font-family-display: "Barlow Condensed", "Inter", ui-sans-serif, sans-serif;
```

### 2. `.stat-num` split into two classes

`index.css` currently has one `.stat-num` (Inter, weight 800, tabular-nums) used for both small inline tabular readouts (agenda dates, ride stats) and large KPI values (StatTile). These are different roles now that a display face exists. Replace with:

```css
/* Inline tabular readouts: agenda dates, ride list stats, small numeric labels.
   Inter stays; weight corrected from 800 (unloaded, synthesized) to 700 (loaded). */
.stat-num {
  font-variant-numeric: tabular-nums;
  letter-spacing: -0.02em;
  font-weight: 700;
}

/* Hero numerals: watts, timers, KPI stat values, session-complete readouts.
   Barlow Condensed display face (D-5). */
.stat-num-hero {
  font-family: var(--font-family-display);
  font-variant-numeric: tabular-nums;
  letter-spacing: -0.01em;
  font-weight: 700;
}
```

Every existing `.stat-num` usage that renders a **hero-scale** value (≥28px — see Typography table) migrates to `.stat-num-hero`. Usages at body/label scale (agenda date column, ride-row stats, week totals) keep `.stat-num`.

### 3. shadcn button token mapping (defect fix)

`components/ui/button.tsx` references `--primary`, `--primary-foreground`, `--destructive`, `--secondary`, `--secondary-foreground`, `--accent`, `--accent-foreground`, `--background`, `--input`, `--ring` — **none of these are defined anywhere in `index.css`**, so every non-`style`-overridden `<Button>` renders on undefined/browser-default values. Add this block to `@theme` in `index.css`, mapped to the existing real palette (no new colors invented):

```css
--color-primary:              var(--color-brand);      /* #1F6FE5 */
--color-primary-foreground:   #FFFFFF;
--color-destructive:          var(--color-bad);         /* #C0341D */
--color-secondary:            var(--color-bg-2);        /* #F6F6F7 */
--color-secondary-foreground: var(--color-ink);
--color-accent:                var(--color-blue-0);     /* #E9F3FC */
--color-accent-foreground:    var(--color-brand);
--color-background:           var(--color-surface);     /* #FFFFFF */
--color-foreground:           var(--color-ink);
--color-border:                var(--color-line);
--color-input:                 var(--color-line);
--color-ring:                  var(--color-brand);
```

After this mapping lands, `<Button variant="default">` alone renders the correct brand-blue fill — remove the redundant inline `style={{ backgroundColor: 'var(--color-brand)', color: '#fff' }}` override on `SessionCard`'s Start button and any other hand-rolled inline CTA as that screen is touched (D-8). `<Button>` becomes the only button component for light-surface screens; raw `<button>` elements on those screens are refactored to `<Button>` as each screen is touched in this phase.

**Exception — dark cockpit:** `DuringSessionScreen`'s in-session controls (Pause/Resume, Skip step, End session, session-complete CTA) stay custom-styled against `--cockpit-*` tokens (below), not `<Button>`. A full dark-mode token remap for shadcn's `<Button>` is Phase 2 scope (DARK-01), not this phase — do not attempt it. This is the one screen where inline-styled controls remain intentional.

### 4. `--color-accent` / hardcoded `#dc2626` in Settings

`SettingsScreen.tsx` has two off-token colors: the "Re-send magic link" button uses `var(--color-accent)` (previously undefined, now defined above as a light-blue tint — **not** the right role for link text) and the sign-out button uses `var(--color-destructive, #dc2626)` (hardcoded hex fallback). Fix:

- Re-send magic link → `var(--color-brand)` (it's an interactive link/secondary action, not destructive). **[Assumption]** — CONTEXT's D-8 groups both fixes under one sentence ending "(--color-bad)"; read literally that would recolor a benign resend-link red, which is a color-semantics regression. This spec resolves the ambiguity as two separate token fixes: resend-link → `--color-brand`, sign-out → `--color-bad`.
- Sign out → `var(--color-bad)` directly (drop the hardcoded hex fallback now that `--color-destructive` is properly defined and equals `--color-bad`).

### 5. Zone map: single source of truth (`lib/zones.ts`)

The zone color/label/percent-range map is currently duplicated 4x: `TodayScreen.tsx` (local `ZONE_VAR`), `AgendaScreen.tsx` (local `ZONE_VAR`, inconsistently alongside an import of `ZONE_META` from `lib/format.ts` in the same file), `DuringSessionScreen.tsx` (local `ZONE_META`), `ZoneChip.tsx` (local `ZONE_VAR` + `ZONE_LABEL`). `lib/format.ts` already has the canonical shape (`ZoneKey`, `ZONE_META`, `zoneColor()`) and `SessionCard.tsx` / `WorkoutProfileChart.tsx` already import it correctly — that is the pattern to generalize.

Action: extract `ZoneKey`, `ZONE_META`, `zoneColor()`, and add `zoneLabel()` into a new `frontend/src/lib/zones.ts`. Re-export from `lib/format.ts` (`export { ZONE_META, zoneColor, type ZoneKey } from './zones'`) so any other existing import path keeps working. Delete the 4 local duplicate maps; all 4 files import from `@/lib/zones` (or `@/lib/format`, which now re-exports it) instead.

Zone hex values are unchanged (already match PRD exactly — verified against `docs/prd.md`):

| Zone | Token | Hex |
|------|-------|-----|
| Recovery | `--color-zone-recovery` | `#2B8A5B` |
| Endurance | `--color-zone-endurance` | `#228BE6` |
| Tempo | `--color-zone-tempo` | `#F0A030` |
| Threshold | `--color-zone-threshold` | `#E8590C` |
| VO2max | `--color-zone-vo2` | `#C92A2A` |

### 6. Shared `PromptChip`

`OnboardingScreen.tsx` and `ChatScreen.tsx` each define an identical local `PromptChip` function (pill button, 8px/14px padding, 999px radius, `--color-line` border, hover swaps `--color-surface` → `--color-bg-2`, 14px `--color-ink-2` text). Extract to `frontend/src/components/ui/PromptChip.tsx` with the exact existing visual treatment (no restyle needed — it's already on-token), both screens import it and delete their local copies.

### 7. `--cockpit-*` tokens (new — scoped dark surface)

Add to `@theme` in `index.css`. This is the **only** part of the app permitted to use these tokens (D-1 exception to light-mode-only MVP, scoped to `DuringSessionScreen`'s active-session route). No pure black.

```css
--color-cockpit-bg:      #14171D;  /* near-black ink surface, NOT #000 */
--color-cockpit-surface: #1D2129;  /* profile rail panel, slightly lifted */
--color-cockpit-ink:     #F2F3F5;  /* primary text on dark, not pure white */
--color-cockpit-ink-2:   #9AA0AC;  /* secondary text on dark */
--color-cockpit-line:    #2A2F3A;  /* dividers on dark */
```

Zone colors are reused as-is on the cockpit (D-7: "must continue to match the PRD hex values exactly" — no cockpit-specific zone variants). **Contrast check required at implementation time:** verify each zone hex maintains ≥3.0:1 against `--color-cockpit-bg` for the large-text uses (zone chip label, hero lozenge fill+white text is already high-contrast). If any zone color fails against the dark bg specifically for small text, lighten via `color-mix(in srgb, var(--color-zone-X) 85%, white)` for that one cockpit instance only — never change the base hex token.

## Typography

Base template roles — capped at exactly 4 distinct sizes, 2 weights (Inter, UI text):

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px | 400 | 1.5 | Body copy, descriptions, chat text |
| Label (micro/eyebrow) | 11px | 600 | 1.4 | Uppercase micro-labels; also the Today-hub date eyebrow above the screen title — same size/weight, sentence case instead of uppercase (renders a full date like "Monday, July 7", not a short label), `--color-ink-3` |
| Heading (card/section titles) | 20px | 600 | 1.2 | Card titles, section headers |
| Display (screen titles, D-10) | 28px | 600 | 1.15 | `AppLayout.tsx`'s `<header>` `<h1>` screen titles; **also** the Today-hub objective heading (`SessionCard`, D-6) — reuses this exact role/size rather than introducing a fifth size. D-6 says "objective to ~28px," which this satisfies directly. Hierarchy between the two is established by context (card body vs. page chrome), not by a size delta. |

"Display weight" per D-10 means Inter 600 at 28px, **not** the Barlow Condensed numeral face — Barlow Condensed is reserved strictly for numerals per D-5. `AppLayout.tsx`'s `<header>` `<h1>` moves from `fontSize:20, fontWeight:700` to 28px/600; the existing per-route date string on Today moves **above** the title as the Label-role eyebrow described in the table (currently rendered below).

Weight budget: 2 Inter weights carry all body/UI text at any one call site (400 regular, 600 semibold — 500/700 exist in the loaded set for legacy call sites but new work should default to 400/600).

### Hero numeral scale — role-scoped exempt secondary system

Barlow Condensed (D-5 — `.stat-num-hero`, or component-local styles using `var(--font-family-display)`) is a **separate, closed set of exactly 3 sizes**, intentionally outside the 4-size UI cap above, because it is not UI text:

1. Different font family entirely, reserved exclusively for numerals per D-5 (never used for words/labels/body copy).
2. Fluid `clamp()` ranges tied to viewport width, not fixed design-system sizes.
3. Distinct role — large-format data readouts (cockpit watts/timer, KPI tile values), analogous to a chart's data-ink rather than interface chrome.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Cockpit primary (watts / effort word) | `clamp(96px, 18vw, 160px)` | 700 | 1 |
| Cockpit timer (demoted secondary) | `clamp(48px, 10vw, 72px)` | 600 | 1 |
| Today-hub StatTile value | `clamp(34px, 8vw, 52px)` (unchanged size, font swapped from Inter 800 → Barlow Condensed 700) | 700 | 1 |

No hero size may be added to this set without a spec update. **Session-complete hero readout** reuses the **Cockpit timer** clamp's upper bound directly — fixed `72px`, Barlow Condensed 700, line-height 1 — instead of introducing a fourth hero size; session-complete is a static screen with no viewport-fit concern, so the clamp's max is used as a flat value rather than the full clamp.

Barlow Condensed carries 2 weights (600/700) exclusively for hero numerals — same "2 weights" discipline as the base table, honoring D-5's explicit amendment for the display face.

Inline tabular readouts (`.stat-num`, Inter, pre-existing sizes unchanged by this phase — agenda date column, ride-row stats, week totals, all 12-13px in production today): weight corrected 800 → 700 per the Foundation Fixes section above. These predate this redesign and are not part of either sizing table above; this phase only fixes their weight, not their size.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--color-bg` `#F2F4F7` | App background across all light-mode screens |
| Secondary (30%) | `--color-surface` `#FFFFFF` (cards) + `--color-ink`/`--color-ink-2` (text) + `--color-line` (borders) | Card surfaces, nav chrome, body/secondary text, dividers |
| Accent (10%) | `--color-brand` `#1F6FE5` | Primary CTA fills, active nav pill (bottom tab + desktop sidebar), links, focus rings |
| Destructive | `--color-bad` `#C0341D` | Destructive actions only (Mark missed confirm, Sign out, delete affordances) |

Accent reserved for: primary "Start ride" CTA fill, active bottom-tab pill, active desktop-sidebar pill, inline text links (e.g. "Back to sign in", chat prompt focus), form focus rings. Never used for informational/decorative fills — those belong to the zone-color system below.

**Zone color is a separate semantic layer, not part of the 10% accent budget.** D-7 intentionally pushes zone-color usage broad and wide (agenda intensity bars, cockpit accents, profile rail, workout profile chart) — this is data encoding (like a chart legend), analogous to how a stock chart uses green/red freely without those being "the brand accent." Treat zone colors (`--color-zone-recovery/endurance/tempo/threshold/vo2`) as an independent, unlimited-budget system reserved for: session/block intensity indicators, wherever a specific training zone is being communicated. Do not reach for zone colors for anything that isn't zone-semantic (e.g. don't use `--color-zone-endurance` as a generic "blue" for unrelated UI — that's `--color-brand`'s job).

**`--color-achieve` (`#F76707`) is a third, single-purpose reserved accent** (D-11): used exclusively on the session-complete screen — the "Back to today" primary CTA fill switches from `--color-blue-6` to `--color-achieve` on that screen only. No other screen or component may use it. This is the "single sanctioned celebration moment" per D-11; do not extend it to streaks, badges, or other future gamification without a new design decision.

Contrast rules (binding, from PRD, unchanged):
- `#228BE6` (blue-6) is 3.56:1 on white — large text, buttons, and fills only, never small body text.
- Small blue text uses blue-7 `#1B73C0` (4.95:1).
- Body copy uses `--color-ink` or `--color-ink-2`.

Cockpit exception (scoped to `DuringSessionScreen` only): background `--color-cockpit-bg`, text `--color-cockpit-ink`/`--color-cockpit-ink-2`, dividers `--color-cockpit-line`. No pure black anywhere, including here.
