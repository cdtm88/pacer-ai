<objective>
Wire the durable-audit + cross-turn-seeding runtime on top of Plan 01's storage layer. Three coupled changes that all live in the agent runtime / SSE chain:

1. Thread `conversation_id` through `sse_generator -> run_turn -> dispatch_tool`, mirroring the `user_id` chain 260702-wev already established (RESEARCH Pitfall 2: it is not threaded today).
2. Make `dispatch_tool` persist one `audit_log` row per tool call via `write_audit_entry` (TRUST-06 durable write), keeping the in-memory `audit_log` list that same-turn injection depends on.
3. Seed `tool_result_values` at the start of every `run_turn` from `load_prior_audit_values`, killing cross-turn false positives on the stateless Vercel backend (TRUST-09 / D-04).

Purpose: TRUST-06 (wiring) + TRUST-09; makes TRUST-04 "verifiable in logs" real end-to-end.
Output: threaded signatures across five backend files + a cross-turn-seed regression test.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client-supplied conversation_id → audit reload | a conversation_id must be scoped to the authenticated user before its tool-result numbers seed the scanner |
| serverless invocation N → invocation N+1 | cross-turn trust state must survive statelessly via Postgres, not an in-process cache |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-08-04 | Tampering (of trust state) | stale/incoherent in-process cache on serverless | medium | mitigate | Seeding reads Postgres per invocation (D-04); no in-memory cache is introduced |
| T-08-01 | Tampering | audit trail faithfulness for laundering detection | high | mitigate | Audit rows record the post-injection inputs actually passed to the function, so the durable trail cannot be spoofed by LLM-supplied argument values (foundation Plan 06 builds on) |
| T-08-SC | Tampering | package installs | low | accept | No new packages; reuses backend.agent.audit + existing SDK chain |
