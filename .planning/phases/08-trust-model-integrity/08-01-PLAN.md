<objective>
Build the durable persistence layer that makes TRUST-04's "verifiable in application logs" literally true for the first time, and provide the read side D-04's cross-turn seeding needs.

This plan creates the `audit_log` Postgres table (plus the `profiles.hr_zones_available` column that Plan 07 will write), and a new `backend/agent/audit.py` module exposing `write_audit_entry` (best-effort per-dispatch write) and `load_prior_audit_values` (conversation-scoped reload). It does NOT yet wire these into `dispatch_tool`/`run_turn` — that is Plan 05. This is the interface-first foundation both Plan 05 (writes + seeding) and Plan 07 (hr_zones_available column) depend on.

Purpose: TRUST-06 storage layer; the single persistence mechanism that serves both D-01 (audit trail) and D-04 (cross-turn number seeding) per RESEARCH.md's finding that the `messages` table is NOT a viable reload source.
Output: One migration file, one new module, one new test file.
</objective>
