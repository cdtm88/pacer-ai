<objective>
Close the one blocking UAT gap in Phase 8: ONBD-05 Branch A (user states their LTHR directly in onboarding) deterministically fails with a trust_violation -> max_retries and returns an empty assistant response.

Root cause (08-UAT.md Test 1 + .planning/debug/onboarding-lthr-selfreport-trust-violation.md): backend/agent/trust.py's `_is_attributed` recognizes ONLY `tool_result_values` (this-turn dispatch_tool results + TRUST-09's cross-turn `load_prior_audit_values`) as a legitimate attribution source. Branch A never calls a tool by design (ONBOARDING_SYSTEM_PROMPT: "use the user's stated LTHR value directly ... no tool call needed"), yet the mandatory D-03 confirmation-gate summary forces the assistant to restate the user's own number ("165 bpm"). scan_buffer flags that restatement as unsourced on every attempt because the number was never a tool result.

This builds the missing half of D-02's original design: a "confirmed values registry populated from actual tool results AND the onboarding profile / self-report." Only the tool-results half (D-04 / TRUST-09) was ever built. This plan adds the self-report half: a per-turn attribution channel extracted from the user's OWN chat messages, with the exact same numeric-token + tolerance rigor already used for tool results.

Purpose: Make Branch A satisfy its D-05 contract without weakening the trust-model invariant (LLM never emits physiological numbers from its own reasoning). The fix attributes a number ONLY when it appears verbatim (within tolerance) in a genuine user-authored message — never opening a loophole where the LLM restates an unsourced number of its own invention.

Output: A distinct `self_reported_values` attribution channel in `scan_buffer`, a pure `collect_self_reported_values` extractor, per-turn wiring in `run_turn`, and regression-proof test coverage (Branch A passes; hallucinated numbers still fail).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user chat message -> trust scanner | Untrusted user-supplied text becomes a numeric attribution allowlist for what the assistant may restate. |
| LLM assistant text -> trust scanner | Untrusted model output is scanned for unsourced physiological numbers before it can reach the SSE stream. |
| LLM/user tool arguments -> dispatch_tool | Computed-output fields (current_ctl / ftp_watts / load_targets) cross here and are already server-side-injected (D-02 / Plan 06); this plan does NOT touch that boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-08-01 | Tampering/Elevation | self_reported_values attribution channel (trust.py) | high | mitigate | The self-report channel only governs what the SCANNER permits the assistant to RESTATE; it never authorizes a tool's computed-output fields. current_ctl / ftp_watts / load_targets remain server-side-injected in dispatch_tool (D-02 / Plan 06) and are never read from LLM/user args, so a number laundered into chat cannot reach a computed field. save_profile's lthr_estimate (Branch A) is the only sink and is exactly where the user's own claim already flowed. |
| T-08-08-02 | Tampering (hallucination pass-through) | scan_buffer attribution | high | mitigate | Attribution requires the number to appear verbatim (within NUMERIC_TOLERANCE, boundary-aware _NUMERIC_TOKEN) in a genuine user message or tool result. A hallucinated number absent from both is still flagged — enforced by the negative-control tests in both Task 1 (unit) and Task 2 (loop). No substring/decimal bypass (reuses the D-03 / TRUST-08 boundary logic). |
| T-08-08-03 | Spoofing (echo->source laundering) | collect_self_reported_values (trust.py) | medium | mitigate | Extraction filters role=="user" string content ONLY; assistant messages and tool-result block lists are excluded, so the assistant's own prior (now-permitted) echo can never become an attribution source on a later turn — the laundering chain is structurally broken. |
| T-08-08-04 | Information Disclosure | matched_text handling | low | accept | No new surfacing of matched numbers; existing scan_buffer / loop redaction invariants (matched_text never reaches an SSE data frame) are preserved unchanged. Rationale: no code path in this plan emits matched_text to the user. |
| T-08-08-SC | Tampering (supply chain) | dependencies | low | accept | No new npm/pip/cargo installs; the change is pure-Python edits to existing modules and tests. No package-legitimacy surface introduced. |
</threat_model>
