  <name>Task 1: ZWO-05, real Zwift import acceptance test</name>
  <action>
    Human-verify checkpoint: a developer generates a .zwo file from the export flow and imports it into the real Zwift app, confirming the file is well-formed, carries sportType bike with no Cadence, uses decimal FTP-fraction Power for FTP sessions and FreeRide+textevent for pre-FTP sessions, and imports without error. No source files change in this task; the executor records the result and, if the import fails, captures the Zwift error and offending XML for a fix to api/sports_science/zwo.py. See how-to-verify for the exact steps.
  </action>
  <what-built>
    The backend export endpoint GET /sessions/{id}/export.zwo (plan 01) and the frontend ZwoExportModal "Download .zwo" flow (plan 04) generate a .zwo workout file from today's session. Power values are FTP fractions (0.0-2.0); pre-FTP sessions use FreeRide segments with textevent RPE cues; the file carries sportType bike and omits Cadence.
  </what-built>
  <how-to-verify>
    1. Start the app (frontend dev server + API) and sign in. Ensure today has a planned structured session (or use the "Ride anyway" path to reach a session).
    2. On Today, tap "Export to Zwift", then "Download .zwo". Confirm a file named like 2026-06-21-endurance.zwo downloads.
    3. Open the file: confirm it is well-formed XML, the root is <workout_file>, it contains exactly one <sportType>bike</sportType>, and no Cadence attribute appears anywhere.
    4. For a profile WITH an FTP: confirm <SteadyState> segments with Power values that are decimals between 0.0 and 2.0 (e.g. 0.5, 0.65), not raw watts and not bare integers.
    5. For a profile WITHOUT an FTP (ftp null): confirm <FreeRide> segments each containing a <textevent> RPE cue, and zero <SteadyState> elements.
    6. Import the file into the real Zwift app (Workouts -> custom/import, or drop into the Zwift workouts folder). Confirm Zwift imports it without error and the workout preview shows the expected number of segments at the expected intensities.
    7. If import fails: capture the Zwift error and the offending XML; this resolves research assumptions A1 (Power is FTP fraction), A2 (textevent child of FreeRide), A3 (sportType "bike") and tells the executor exactly what to fix in api/sports_science/zwo.py.
  </how-to-verify>
  <acceptance_criteria>
    - A .zwo file downloads with filename pattern {YYYY-MM-DD}-{type}.zwo
    - The file imports into the real Zwift app with no error
    - Zwift shows the correct segment count and intensities (SteadyState for FTP sessions, FreeRide+text for pre-FTP)
    - Human types approval confirming the import succeeded
  </acceptance_criteria>
  <resume-signal>Type "approved" if the .zwo imported cleanly in Zwift with correct structure, or describe the import error and the offending XML so the executor can fix api/sports_science/zwo.py.</resume-signal>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: IOS-03, during-session timer + wake lock on physical iOS Safari</name>
  <action>
    Human-verify checkpoint: a developer opens the deployed app on a physical iPhone, installs it to the Home Screen, runs a during-session timer, and confirms the screen stays lit, the timer resyncs after backgrounding, auto-advance and Skip and the Session complete overlay all work, and records the tested iOS version. No source files change in this task; the executor records the result and, if anything fails, captures the symptom and iOS version for a fix to the hooks. See how-to-verify for the exact steps.
  </action>
  <what-built>
    DuringSessionScreen (plan 03) composes useSessionTimer (Date.now() delta countdown with visibilitychange resync) and useWakeLock (native Wake Lock with NoSleep.js fallback for iOS < 18.4). The screen auto-advances steps, shows a 3-second countdown warning, supports Skip, and shows a Session complete overlay.
  </what-built>
  <how-to-verify>
    1. Deploy the frontend (or expose the dev server over HTTPS / via the Vercel preview) so it is reachable on a physical iPhone. Wake lock requires HTTPS.
    2. On the iPhone in Safari, open the app and add it to the Home Screen (Share -> Add to Home Screen). Launch it from the Home Screen icon (installed PWA mode - this is where the iOS < 18.4 wake-lock bug appears, per RESEARCH Pitfall 1).
    3. Start a session. Confirm the timer counts down in MM:SS and the current step label is large and legible.
    4. Leave the screen untouched for at least 60 seconds. Confirm the screen does NOT dim or lock (wake lock / NoSleep.js holding it lit). Note the iOS version.
    5. Switch to another app for ~20 seconds, then return. Confirm the timer has advanced by roughly the elapsed wall-clock time (resynced via visibilitychange), not frozen and not reset to the step start.
    6. Let a step reach its end. Confirm the "Starting [next step] in 3..." countdown appears and the screen auto-advances. Tap "Skip step" on a step and confirm it advances immediately.
    7. Complete the last step. Confirm the "Session complete" overlay shows total time, "{N} steps completed", and "Back to today" returns to Today.
    8. Record the tested iOS version. If the screen dims on an iOS < 18.4 device, confirm whether the NoSleep.js fallback engaged (it may require an initial tap/gesture per RESEARCH assumption A4).
  </how-to-verify>
  <acceptance_criteria>
    - Screen stays lit for at least 60 seconds during an active session on a physical iOS device (Home Screen installed mode)
    - After backgrounding ~20s and returning, the timer reflects elapsed wall-clock time (resync, not reset, not freeze)
    - Auto-advance, 3-second countdown warning, Skip, and Session complete overlay all work on iOS Safari
    - The tested iOS version is recorded
    - Human types approval confirming the iOS behavior
  </acceptance_criteria>
  <resume-signal>Type "approved" with the tested iOS version if the timer and wake lock work correctly on physical iOS Safari, or describe what failed (screen dimmed, timer froze/reset, auto-advance broken) so the executor can fix the hooks.</resume-signal>
</task>
