<objective>
Fix a cross-round trust-attribution false positive in the shared agentic loop. In `backend/agent/loop.py`, `run_turn` declares `tool_result_values: list[str] = []` INSIDE the `while retries <= MAX_RETRIES` loop, so it is reset on every iteration. Because that loop body is reused for BOTH trust-retries AND normal tool-use continuation rounds, the final text-only round (the summary that describes zones/plan in prose) scans against an EMPTY `tool_result_values` — every physiological number sourced from a tool call in an EARLIER round of the same turn gets flagged as unattributed. This trips repeated `TrustViolation` errors and `max_retries (3) exceeded`, blocking every onboarding confirmation from completing in production even after the 260702-vsp Pattern A attribution fix shipped.

The fix: move the single line `tool_result_values: list[str] = []` OUT of the while loop, initialized once alongside `retries` and `tool_turns`, so it accumulates across all iterations of one `run_turn()` call. `text_buffer` and `seen_calls` stay reset per iteration (intentional, unrelated).

Purpose: Unblock multi-round tool-use conversations (onboarding confirmation, regular chat) from false trust violations.
Output: One-line relocation in `loop.py` + one new TDD regression test in `test_loop.py`, committed and pushed to main (auto-deploys the Vercel Python function).
</objective>
